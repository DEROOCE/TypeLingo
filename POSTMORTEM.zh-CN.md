# TypeLingo 第一阶段项目复盘

这份文档总结了 TypeLingo 第一阶段的开发过程。这个阶段的核心产物，是一个基于 macOS 的实时字幕翻译原型：它会监听当前聚焦的输入框，在接近实时的条件下翻译输入内容，并把结果显示在一个悬浮字幕浮窗中。

这个阶段的目标并不是把产品一次性做完，而是回答一个更关键的问题：

我们能不能先做出一个真正可运行的 macOS 工具，稳定地从任意输入场景中拿到文本，接入可配置的翻译后端，再以字幕浮窗的形式把结果展示出来，并让整条链路在真实使用中成立？

结论是：可以，但有一个明确边界。当前原型已经足够成为一个真实可用的本地工具，但它还不是最终技术形态。最大未解问题仍然是中文等 IME 输入法的组合态文本捕获，而这件事最终会要求项目从基于 Accessibility 的监听方案，升级到基于 `InputMethodKit` 的输入法方案。

## 1. 项目背景

TypeLingo 的出发点很直接：当用户在 macOS 上任意输入框里打字时，实时把这些文字翻译成目标语言，并显示成一个悬浮字幕。

这个想法从产品表面看起来很轻，但底层实现并不轻。

从产品层看，它像是一个“翻译字幕浮窗”。但从系统层看，它本质上是一个 macOS 输入捕获问题。真正难的地方不是翻译 API，而是这个工具能不能在不同应用、不同输入控件、不同输入法条件下，可靠地拿到用户当前正在输入的文本。

这一点，决定了整个项目的技术路线和阶段目标。

## 2. 第一阶段目标

第一阶段主要有五个目标：

1. 在不一开始就进入完整输入法开发的前提下，验证 macOS 上“监听当前聚焦输入框”的可行性。
2. 做一个真正的悬浮字幕工具，而不是只在应用内部演示的假 demo。
3. 至少支持两类翻译后端，避免产品被单一 provider 锁死。
4. 把配置能力做到足够实用，包括 provider、prompt、target language、快捷键、外观等。
5. 把项目整理成一个可以被他人查看、运行、继续迭代的开源仓库。

这个阶段明确不打算一次解决所有平台难题。尤其是不打算在第一阶段彻底解决：

- 中文 IME 输入中间态
- 所有应用场景下的稳定兼容
- 面向公众分发所需的正式签名和 notarization

## 3. 第一阶段实际完成了什么

到第一阶段结束时，项目已经从一个粗糙想法演进成了一个结构完整、可运行、可开源的 macOS 工具，具备以下能力：

- 基于 SwiftUI 和 AppKit 的 macOS 原生应用
- 基于 Accessibility API 的聚焦输入监听
- 可独立显示和缩放的实时字幕浮窗
- 可配置的全局唤醒快捷键
- `Google Web` 翻译支持
- `OpenAI-compatible` 翻译支持
- 多个 API profile
- 多个 prompt profile
- 设置导入导出
- API key 存储到 macOS Keychain
- 本地打包为 `.app`、`.zip`、`.dmg`
- 开源仓库整理和第一版 GitHub Release

这意味着项目已经不是一个单纯的 UI 演示，也不是单一功能拼出来的脚本集合，而是一条完整的、可以反复运行的产品链路：

输入监听 -> 文本提取 -> 翻译调度 -> 字幕渲染 -> 配置管理 -> 打包分发

## 4. 开发过程中的核心思考

### 4.1 最重要的第一个决策：先做对的原型

最早期最重要的选择，不是 UI 怎么做，也不是该用哪家大模型，而是：到底应该做哪一层原型。

当时其实有两条路：

- 一上来直接做最终架构，也就是 `InputMethodKit`
- 先用风险更低的 `Accessibility API` 做出完整产品链路，验证交互和使用价值

最终选择了第二条路。

这背后的原则很简单：在进入最重的系统级开发之前，先验证整条产品闭环是否真的成立。对 TypeLingo 来说，这条闭环就是：

聚焦输入 -> 文本捕获 -> 翻译 provider -> 字幕浮窗 -> 配置和调度

事实证明，这是正确决策。它让项目很快进入“真实可用”的状态，也更早暴露出真实问题，而不是困在一开始就最复杂的系统集成里。

### 4.2 尽早识别真正的瓶颈

整个项目里，一个很关键的判断是：翻译从来都不是最难的部分。

真正难的是输入捕获：

- 不同应用框架暴露文本能力不一样
- Secure input 不能碰
- Electron、游戏、远程桌面行为不一致
- 中文 IME 组合态文本通常无法被当前的 Accessibility 轮询稳定拿到

这个判断非常重要，因为它决定了项目该优先优化哪一层。如果一开始把它当成“一个翻译 UI 工具”，那很多精力就会被浪费在错误的层次上。

相反，这次开发里更优先关注的是：

- 系统监听能力
- 文本流转链路
- 悬浮窗可用性
- 配置和状态管理

而不是先做大而全的品牌化 UI 包装。

### 4.3 快速验证和长期正确性要分开

整个过程中还有一个持续遵守的原则：不要把原型包装成已经解决了最终问题。

所以有几件事一直保持明确：

- Accessibility 方案适合原型，但不等于真正输入法方案
- 本地可运行的 ad-hoc 包，不等于正式可分发版本
- 明文存储 secret 只可能是过渡状态，不能作为正式设计

这种做法的价值在于：项目可以跑得很快，但不会因为掩盖根本问题而在后期付出更高成本。

## 5. 开发中的主要难点，以及如何处理

### 5.1 难点一：macOS 上的“任意输入框监听”

这是整个项目的基础难点。

用户需求是“任意输入框”。但在 macOS 上，这并不是一个统一接口。系统要面对的是：

- 原生 AppKit 文本控件
- 浏览器输入框
- Electron 应用
- 自定义文本组件
- 根本不暴露文本能力的控件
- 必须跳过的安全输入框

第一阶段采用的方案，是通过 Accessibility API 去轮询当前聚焦输入元素。这让项目能快速拿到真实文本，并在很多标准输入框场景下表现可用。

但这条路也有明确限制：

- 足够验证产品价值
- 不足以稳定覆盖 IME 组合态文本

这次没有试图掩盖这个问题，而是直接把它收敛为下一阶段的核心技术任务：如果 TypeLingo 要真正稳定地覆盖中文输入，就必须转向 `InputMethodKit`。

### 5.2 难点二：浮窗容易长成“调试面板”

字幕浮窗一开始很容易变成一个把各种状态、按钮、菜单都堆上去的工具面板。

这种倾向会带来几个直接问题：

- 看起来不像字幕产品，而像开发中的工具窗
- 信息噪声太大
- 长文本可能显示不稳
- 技术上“能点”的控件不一定真的值得保留

这次的处理方式是持续收敛：

- 从浮窗里去掉不必要的原文显示
- 压缩状态信息
- 让浮窗只专注于翻译字幕本身
- 把复杂操作转移到设置面板
- 保留少量高频操作在浮窗顶部
- 支持自由缩放，让字幕本身优先可读

这里最大的经验是：浮窗本质上应该像“字幕卡片”，而不是“设置容器”。

### 5.3 难点三：配置系统很快变复杂

项目初期看起来只是几个简单设置，但很快就扩展出了完整配置系统：

- target language
- 当前 provider
- 多个 provider profile
- 多个 prompt profile
- 字幕字号
- 背景透明度
- 唤醒快捷键
- 导入导出

到了这个阶段，配置已经不再是“顺手存在 UserDefaults 里”的小事，而变成了一个独立子系统。

这里踩过的最大坑，是敏感信息处理。最初 API key 会落到 preferences 里，这在正式产品里是不合格的。

后续处理方式是：

- 把 API key 移到 Keychain
- 非敏感 profile 数据保留在 preferences
- 导出设置默认不带 key
- 导入导出支持迁移和兼容

这是项目从“原型能跑”向“配置模型基本靠谱”迈出的关键一步。

### 5.4 难点四：打包和分发不是最后顺手做一下

打包这件事一开始看上去像是尾声工作，实际不是。

项目在这部分遇到的问题包括：

- `.app` 是否真的能启动
- 本地 ad-hoc 签名和无签名的差别
- 为什么 Accessibility 授权会在反复打包后失效
- `.zip/.dmg` 能不能给别人用
- Gatekeeper 为什么会拦
- 什么是 Developer ID、什么是 notarization

最后的处理方式，是把“本地可运行”和“正式可分发”严格拆开：

- 本地 `.app` 用 ad-hoc 签名，保证自己机器上好启动
- release 脚本可以生成 `.zip` 和 `.dmg`
- 文档和 release note 明确写清楚：未 notarized，不适合正式对公众分发

这个部分很不显眼，但对真实项目非常重要。因为用户拿到的不是代码，而是包。

### 5.5 难点五：前期过于依赖人工验证

在项目早中期，大部分问题都只能靠手工点、手工运行、手工观察。

对 UI 迭代来说，这没有问题。但当项目开始涉及：

- 配置迁移
- secret 脱敏
- 导入导出
- 打包脚本

纯手工验证就开始变得危险。

所以后来补上了一小套自动化测试，重点覆盖：

- secret sanitization
- 导出行为
- 快捷键模型
- 从明文配置迁移到 Keychain

测试量不大，但足以提高后续重构的信心。

## 6. 哪些地方做得比较好

### 6.1 MVP 边界选得对

第一阶段没有试图一次解决所有问题，而是优先把“真实产品链路”做出来，同时保留对平台级难题的诚实判断。

这是一个非常重要的成功点：原型是真实的，但没有假装它已经是终局架构。

### 6.2 迭代速度非常高

这次开发过程中，围绕真实使用反馈做的迭代非常快。

无论是：

- 浮窗交互
- 设置面板
- provider 切换
- prompt profile
- 快捷键录制
- 导入导出
- release 打包

都在持续、高频地被调整。这种速度对原型阶段是非常有价值的，因为很多产品细节只有在真实跑起来以后才会暴露。

### 6.3 产品表面已经开始变得统一

到第一阶段结束时，TypeLingo 已经不再像一组零散功能，而是逐渐形成了统一的产品表面：

- 清晰的主路径
- 比较明确的字幕浮窗形态
- 成熟起来的 provider abstraction
- 足够实用的设置模型
- 能被公开查看和使用的开源仓库

### 6.4 没有回避最难但最重要的问题

这次最大的成功之一，是项目结束第一阶段时，对“真正没解决的问题”比一开始更清楚，而不是更模糊。

这个问题现在已经被很明确地收敛为：

如何在 macOS 上稳定拿到 IME 输入组合态文本

这是一种健康状态。说明下一阶段不是继续模糊试错，而是明确进入一个核心技术验证问题。

## 7. 哪些地方还可以做得更好

### 7.1 阶段目标应该更早文档化

这次项目中途其实经历了三种不同状态：

- 快速原型
- 个人可用工具
- 公开开源项目

这三者相关，但目标不同。如果更早写一页“当前阶段说明”，很多中途的方向切换和额外整理成本会更低。

未来建议每一阶段开始前至少写清楚：

- 这一阶段必须证明什么
- 这一阶段明确不解决什么
- 什么时候算完成

### 7.2 命名收口偏晚

后续从 `Live Translate` 统一迁移到 `TypeLingo` 时，涉及到了：

- App 名称
- 可执行文件名
- 打包脚本
- bundle id
- release 资产命名
- 仓库文档

这说明命名在“要公开出来”之前就应该尽早定下来。越往后改，成本越高。

### 7.3 `AppState` 承担了过多职责

当前实现里，`AppState` 的价值在于推进快，但它承担的职责已经过多：

- UI 状态
- 持久化
- 配置迁移
- 导入导出
- provider/profile 选择
- 快捷键管理
- 错误状态

对原型来说可以接受，但这是进入第二阶段前最明显的结构风险点。

### 7.4 自动化测试补得有点晚

虽然现在已经有了一套关键路径测试，但理想情况下，这类测试应该更早进入项目，尤其是在：

- provider profile 成型时
- 导入导出完成时
- Keychain 迁移设计确定时

这样很多后续重构会更轻松。

## 8. 开发效率复盘

从整体上看，这次开发效率是比较高的。

因为这个项目同时覆盖了很多层面：

- 系统监听
- 原生 UI
- 翻译 provider
- 配置系统
- secret 存储
- 快捷键
- 打包和分发
- 开源仓库整理
- GitHub Release

在这样的复杂度下，第一阶段还能形成一个完整闭环，说明整体推进节奏是有效的。

但如果要继续做下一阶段，效率上仍然有几个明显可提升点。

### 8.1 一开始就写一页简化版设计说明

不需要完整架构文档，但至少要有一页 Phase Brief，包括：

- 当前阶段目标
- 目标用户使用结果
- 平台限制
- 明确的非目标
- 验收标准

这个投入很小，但能大幅减少中途反复校准方向的成本。

### 8.2 更早拆分状态和配置边界

第二阶段开始前，更合理的拆法应该是：

- runtime app state
- persisted settings
- provider profile store
- prompt profile store
- secrets store
- overlay appearance state

这样后面无论继续加功能还是继续重构，都会更稳。

### 8.3 尽快补最小 CI

至少应该自动跑：

- `swift build`
- `swift test`

这部分投入低，回报很高。尤其是现在仓库已经公开，CI 是把项目从“个人目录”升级成“可维护仓库”的基础设施。

### 8.4 把 `InputMethodKit` 当成独立研究轨

下一阶段最重要的技术风险，不应该混在日常 feature 开发里一起推进。

更合理的做法是单独做一个技术验证：

- 最小输入法原型
- 验证组合态文本捕获能力
- 评估会带来哪些新的用户体验约束
- 看现在的 overlay 和 provider 逻辑能复用多少

这样可以避免把第二阶段拖进一个“大功能泥潭”。

### 8.5 发布流程还可以再标准化

现在打包和 release 已经能工作，但还不够标准化。下一步建议补：

- GitHub Actions
- release checklist
- tag 驱动的 release 自动化
- 如果以后有 Apple Developer 账号，再补正式 notarization 流程

## 9. 当前项目状态判断

到第一阶段结束时，TypeLingo 可以被准确地描述为：

- 一个真实可运行的 macOS 原型
- 一个已经具备日常可用性的本地工具
- 一个方向明确的开源仓库
- 但还不是最终技术架构

这是一个很健康的阶段结果。

现在项目不是“不知道该做什么”，而是已经把最大的不确定性收敛成了一个明确问题。对一个第一阶段原型来说，这就是很好的结束方式。

## 10. 第二阶段建议重点

下一阶段建议只聚焦三件事：

1. 做 `InputMethodKit` 的最小可行性验证
2. 在代码层拆分状态与配置边界
3. 建立最小 CI 和基础发布规范

其他事情都应该排在这三件事后面。

## 11. 最终评价

第一阶段最重要的成果，并不是“做出了一个可以打包的 macOS App”。

更重要的是，这个阶段已经完成了以下几件事：

- 验证了产品主链路成立
- 明确了平台约束到底在哪里
- 建立了一个可以持续迭代的代码库
- 完成了打包、开源和 release 的基础动作
- 把下一阶段最重要的技术问题明确收敛出来

这意味着第一阶段是成功的。

项目没有在做到一半时发现问题定义错了，也没有把最难的问题藏起来。相反，它用一个真正能运行的原型，把产品方向和技术边界都拉清楚了。

# 小红书笔记草稿

## 标题

做完这个实时翻译字幕工具，我终于明白：难的根本不是翻译

## 正文

最近我做了一个 macOS 小工具，叫 TypeLingo。

它要做的事情听起来很简单：
监听当前输入框里的文字，实时翻译成目标语言，然后显示成一个悬浮字幕浮窗。

一开始我以为最难的是接 Google 翻译或者大模型 API。真正做下来才发现，最难的根本不是翻译，而是“怎么稳定拿到用户正在输入的内容”。

这次开发里，我最深的几个感受是：

1. 系统级产品，难点通常不在表面功能。
这个项目表面上像一个字幕浮窗，底层其实是 macOS 输入监听问题。不同应用、不同控件、不同输入法，行为完全不一样。尤其是中文输入法的组合态文本，根本不是普通输入监听能稳拿到的。

2. 原型阶段最重要的，不是一步做到终局，而是先验证完整链路。
所以我没有一开始就冲最重的输入法架构，而是先用 Accessibility API 做出一版能真正跑起来的原型，把“输入监听 -> 翻译 -> 浮窗展示 -> 配置管理”先打通。这一步非常值，因为很多问题只有跑起来才会暴露。

3. 浮窗类产品，体验细节比功能堆砌更重要。
比如按钮该不该留、原文该不该显示、字幕太长会不会被遮掉、透明度怎么调才不影响阅读、快捷键怎么设置才符合直觉。这些看起来都是小问题，但它们决定了这个工具到底像产品，还是像调试面板。

4. 配置和安全不能等到最后再补。
项目后面很快就有了多个 provider、多个 prompt、导入导出、快捷键、字体大小、透明度这些配置。如果一开始不整理，后面会非常乱。后面我又把 API key 从普通配置迁到了 macOS Keychain，这一步其实很关键。

5. 打包和分发也是产品的一部分。
很多人会觉得功能做完再打包就行，但 macOS 上根本不是这样。签名、权限、Gatekeeper、notarization，每一项都会直接影响别人能不能真正用起来。

目前这个项目已经完成了第一阶段：

- 可以监听多数标准输入框
- 可以实时翻译并显示字幕浮窗
- 支持 Google Web 和 OpenAI-compatible provider
- 支持多个 API profile 和 prompt profile
- 支持快捷键唤醒、设置导入导出、Keychain 存储密钥
- 已经开源

但我也更清楚它下一阶段真正要解决的问题了：
不是继续堆更多功能，而是把底层输入捕获能力升级到更稳的输入法级方案。

做完这一轮，我最大的感受就是：
很多“看起来很轻”的工具，真正难的都藏在系统边界里。

如果你也在做这类桌面工具，我会非常建议你先问自己一个问题：
你现在在优化的是产品表面，还是那个真正决定项目生死的底层能力？
